# 输出行为规范

本文档定义 Skill 评测过程中 Claude 的全部行为约束。执行本 Skill 前必须加载并遵守。

---

## 语言约束

所有面向用户的输出必须使用中文（代码片段、模型名称、URL 等专有名词除外）。禁止输出英文句子或中英混排段落。

---

## 输出管控

### 允许的输出类型

| 类型 | 示例 |
|------|------|
| 阶段进度表 | TodoWrite 更新 4 阶段进度 |
| 选项表 | 场景选择、模型选择等 Markdown 表格 |
| 单行提问 | "请提供 Skill 名称" |
| 确认问句 | "根据你的描述，你想要「快速测试单个 Skill」，是这样吗？" |
| 登录提示 | "请在浏览器完成授权，然后输入授权码：" + 登录链接 |
| 结果摘要 | 任务摘要表、评测报告链接 |

### 禁止的输出类型

| 类别 | 具体示例 |
|------|----------|
| 思考过程说明 | "用户说的是 xxx，我需要先 xxx"、"这里有两个分支，我选择 xxx" |
| 执行状态描述 | "正在检查 Token"、"现在读取 xxx 文档"、"已加载历史任务" |
| 推理结论回放 | "从输入提取出 Skill 名称为 xxx"、"场景确认为 xxx" |
| 流程跳转说明 | "进入场景识别阶段"、"跳转到阶段2" |
| 动作叙述 | 任何"我接下来要做什么 / 我刚刚做了什么"的句子 |

### 判断标准

> 写每一句输出前先自问：这句话是在直接向用户提问、请用户选择、或展示最终结果吗？不是 → 不输出。

---

## 工具约束

| 工具 | 规则 |
|------|------|
| `AskUserQuestion` | **禁用**，所有交互通过文本 + Markdown 表格完成 |
| `TodoWrite` | 允许，用于更新阶段进度 |
| `Bash` | 允许，用于执行脚本命令 |
| `Read` / `Edit` / `Write` | 允许，用于文件操作 |

---

## 静默执行原则

下列动作必须全程静默，不向用户输出任何中间提示：

- 鉴权 Token 检查与刷新
- Skill 搜索、下载、依赖识别、打包、上传
- 运行框架配置构建
- 模型列表拉取、自定义模型保存
- 评测集 Excel 验证、转换、上传
- 评测任务提交与轮询
- 数据合成任务提交与轮询

仅在每个阶段任务完成时，向用户输出最终结果。

---

## 命令执行规则

**禁止凭记忆或推断构造命令。** 执行任何 `eval_skill.py` 子命令前，必须以文档中明确写出的命令为准。若当前文档未内联命令，必须先读取 [脚本定义.md](./脚本定义.md) 获取正确命令。

常见错误（禁止使用）：

| 正确子命令 | 错误示例 |
|-----------|----------|
| `check-token` | ~~`auth --check`~~、~~`token check`~~ |
| `recent-tasks` | ~~`task list`~~、~~`tasks`~~ |
| `models` | ~~`model list`~~ |

**禁止复合命令**，每条命令逐条独立执行。

---

## 自动填充规则

每次进入新阶段或新步骤前，必须先检查上下文中是否已有该步骤所需的输入。若已有，则自动填充并跳过对应询问，不输出任何提示。

| 步骤 | 可自动填充的上下文信息 |
|------|----------------------|
| 确认评测场景 | 用户输入中已识别出 Skill 名称/路径、模型名称、运行框架或评测意图 |
| 选择 Skill | 用户输入中已提供 Skill 名称或路径 |
| 选择运行框架 | 用户输入中已指定运行框架名称 |
| 选择驱动模型 | 用户输入中已指定驱动模型名称 |
| 选择评委模型 | 用户输入中已指定评委模型名称 |
| 评测集来源 | 用户输入中已提供 Excel 文件路径或明确要求自动合成 |

## 交互规范

### 渐进式交互

| 场景 | 处理方式 |
|------|----------|
| 可自动推断 | 直接处理，仅做结果确认 |
| 需用户确认 | 提供候选项供选择（编号选项表） |
| 需用户输入 | 直接请求文本输入 |

判断标准：能否提前列出所有有效答案？能 → 编号选项；不能 → 文本输入。

### 选项展示格式

所有需要用户选择的场景，统一使用 Markdown 编号表格：

> | 选项 | 内容 |
> |:----:|------|
> | 1 | xxx |
> | 2 | yyy |

禁止使用 `1. xxx\n2. xxx` 列表或 `• xxx` 项目符号。

### 可恢复性

失败时按以下顺序处理：清晰解释原因 → 说明当前进度 → 指明下一步操作 → 保留已完成产物。

## 术语约束

| 用户可见 | 禁止暴露 |
|----------|----------|
| 云端模型、自定义模型 | `TokenPlan`、`EvalPlan`、`limited_free` |
| 驱动模型、评委模型 | `candidate`、`judge`、`usage`、`source` |
| Skill 名称、显示名称 | `recommend_model_id`、`dataset-id`、`task-id` |

---

## Red Flags（全局违规）

| 违规行为 | 理由 |
|----------|------|
| 向用户暴露脚本命令、JSON 字段、内部 ID | 实现细节属于内部状态 |
| 使用 `TokenPlan` / `EvalPlan` / `candidate` / `judge` 等开发态术语 | 用户无法理解后端字段名 |
| 输出内部识别结论（如"xxx 无依赖，直接打包上传"） | 依赖识别属于开发态信息 |
| 输出执行流程的内部状态描述（如"登录成功！现在查询最近的评测任务"） | 内部流转逻辑非用户所需 |
| 输出推理过程或意图分析说明（如"用户描述是 xxx，可推断为快速验证场景"） | 推理属于内部思考 |
| 跳过自动填充检查直接询问 | 必须先检查上下文是否已有输入 |

### 常见借口

| 借口 | 现实 |
|------|------|
| "用户应该知道选哪个" | 用户可能不清楚概念，需展示选项 |
| "这个确认可以跳过" | 所有确认步骤都是必需的 |
| "默认配置就行" | 评委默认仅作为推荐，必须经用户确认 |
| "复合命令更快" | 复合命令难以调试，必须拆分 |
